# Structuring Flyte Projects with uv

## Section 1: The Two Layers

Every Flyte + uv project involves two distinct layers. Understanding this distinction is the foundation for every decision that follows.

**The image** (slow-changing): the Python environment, installed packages, system dependencies, the interpreter. The SDK computes an MD5 hash of the image's layer stack and only rebuilds when a layer actually changes. A well-designed image rebuilds only when dependencies change.

**The code bundle** (fast-changing): your task source code, packaged as a tarball and uploaded on every run. The container downloads and unpacks it at startup. This is what changes constantly during development.

```
+-----------------------------------+
|       Docker Image (slow path)    |
|  Python interpreter               |
|  Installed packages (uv sync)     |  <- rebuilt only when deps change
|  System packages (apt)            |
|  Content-hashed, registry-cached  |
+-----------------------------------+
|       Code Bundle (fast path)     |
|  Your task source files           |  <- uploaded on every run
|  Local library code               |
|  Tarball extracted at startup     |
+-----------------------------------+
```

Keep these two layers separate. Your image definition should describe only the environment. Source code, including local libraries, travels in the code bundle (in fast-deploy mode) or gets baked in at deploy time (in full-build mode). Either way, it does not belong in the image definition during development.

Violating this principle (copying source into the image with `with_source_folder()` or using `install_project` mode for local code) means your image hash changes on every code edit, causing a full Docker build and push on every iteration.

## Section 2: A uv Primer

uv is a fast Python package manager. If you know pip or Poetry, here is what maps to what:

**pyproject.toml** replaces `requirements.txt`, `setup.py`, `setup.cfg`, and `pyproject.toml` (Poetry-style) all in one:

```toml
[project]
name = "my-project"
version = "0.1.0"
requires-python = ">=3.11"
dependencies = [
    "numpy>=1.26",
    "pandas>=2.0",
]

[build-system]
requires = ["uv_build>=0.9,<0.10"]
build-backend = "uv_build"
```

**uv.lock** is generated by `uv lock` and pins every transitive dependency to a specific version and hash. Always commit it. This is what gives image builds their reproducibility; the SDK uses the lock file to compute the image hash.

```bash
uv lock             # generate or update the lockfile
uv sync             # install from the lockfile into a venv
uv sync --group ml  # install only a specific dependency group
```

Always use `uv run python <script>` rather than running `python` directly. This ensures the lockfile exists and the environment is in sync with it before your script runs.

**Dependency groups** let you define subsets of your dependencies by use case:

```toml
[dependency-groups]
etl = ["pandas", "pyarrow"]
ml  = ["torch", "scikit-learn"]
dev = ["pytest", "ruff"]
```

This is important for monorepos: each `TaskEnvironment` can install only the group it needs, producing lean images instead of one fat image with everything.

**Path dependencies** reference local packages on disk:

```toml
[tool.uv.sources]
my-lib = { path = "../my_lib", editable = true }
```

`editable = true` means changes to the source are reflected immediately without reinstalling.

## Section 3: How the Image Gets Built

`flyte.Image` is a frozen, content-addressed layer stack. Each `.with_*()` call appends an immutable layer. The final image tag is an MD5 hash of all layers.

The primary method for uv projects is `.with_uv_project()`:

```python
image = flyte.Image.from_debian_base().with_uv_project(
    pyproject_file=Path("my_app/pyproject.toml"),
)
```

**Two installation modes:**

`dependencies_only` (default): Only `pyproject.toml` and `uv.lock` are included in the build context. The image installs only the declared dependencies (`uv sync --no-install-project`). The image hash covers only these two files. Your code does not affect the image hash.

`install_project`: The entire project directory is copied into the build context and `uv sync` runs a full install including the project itself. The image hash covers all source files. Any code change triggers a full image rebuild. Use this only when you need the project installed as a proper package (e.g., when you need package entry points or compiled extension modules that cannot be installed another way). For most Flyte projects, `dependencies_only` is sufficient. `install_project` is an edge case.

With `dependencies_only`, the build context is kilobytes (just `pyproject.toml` and `uv.lock`). With `install_project` or `with_code_bundle()` + `copy_style="none"`, it includes source files and grows with your codebase.

## Section 4: `with_code_bundle()` -- One Image Definition for Development and Production

`with_code_bundle()` is how you write an image definition that works for both development and production without changing any code. It is a declaration, not an immediate instruction.

```python
image = (
    flyte.Image.from_debian_base()
    .with_uv_project(pyproject_file=Path("pyproject.toml"))
    .with_code_bundle()
)
```

Its behavior depends entirely on `copy_style` at run time:

**Fast deploy (default, `copy_style="loaded_modules"` or `"all"`):**
`with_code_bundle()` is a no-op. The image is unchanged. Source travels as a tarball alongside the run submission, extracted at container startup. The image only rebuilds when `pyproject.toml` or `uv.lock` changes.

**Full build (`copy_style="none"`):**
`with_code_bundle()` resolves to a `COPY` instruction. Source files are baked into the image. The container becomes self-contained. This is your production path.

```python
# Development -- run a task directly, code bundle handles source delivery
flyte.run(my_task)

# Production -- deploy an environment with source baked into the image
flyte.deploy(my_env, copy_style="none", version="1.2.3")
```

The key benefit: same image definition, same code, two deployment modes controlled at run time.

**`"loaded_modules"` vs `"all"`:**

`"loaded_modules"` (default): only files that have actually been imported at bundle-creation time are included. This is efficient but requires that all task code has been imported before `flyte.run()` is called. Works well for single-entry-point projects.

`"all"`: everything under `root_dir` (filtered by `.flyteignore`/`.gitignore`) is included. Slightly larger bundle but safe for monorepos with many packages where not everything is imported from a single entry point.

**When to use `with_source_folder()`:**

`with_source_folder()` always copies files into the image build context, regardless of `copy_style`. For your own application code, avoid it — in fast-deploy mode, this means your image hash changes on every source edit. Use `with_code_bundle()` for your own source.

`with_source_folder()` is appropriate in two cases: (1) non-Python assets that need to be permanently in the image (configuration files, model weights, certificates); and (2) baking in a sibling library's source in Pattern B (independent packages) — see Section 7 for details.

## Section 5: `root_dir`

`root_dir` tells Flyte two things: where to look when building the code bundle, and what path prefix to strip when packaging and extracting files.

When the code bundle is built, Flyte walks `root_dir` and collects all Python source files under it (filtered by `loaded_modules` import tracking or `all`). When the container starts, the bundle is extracted and `root_dir` is added to `sys.path`. **The rule: set `root_dir` to the directory you would `cd` into before running `python -c "import my_module"`.**

**src-layout (recommended):**

```
my_project/
+-- pyproject.toml
+-- uv.lock
+-- src/                  <- set root_dir here
    +-- my_project/
        +-- __init__.py
        +-- tasks.py
```

In your entry point script (e.g., `main.py`):

```python
flyte.init_from_config(
    root_dir=Path(__file__).parent.parent,   # -> src/
)
```

Setting `root_dir` to the project root instead of `src/` causes import errors. Always point it at `src/` for src-layout projects.

**Flat layout:**

```
my_project/
+-- pyproject.toml
+-- uv.lock
+-- tasks.py              <- importable as `tasks`
```

```python
flyte.init_from_config(root_dir=Path(__file__).parent)   # -> my_project/
```

For workspace layouts, `root_dir` is covered in Section 7 alongside each pattern.

## Section 6: Where to Define the Image

Image definitions live in the Python file that defines the `TaskEnvironment`. The `pyproject_file` path is relative to where the image builder runs, which is always the local filesystem at `flyte run` / `flyte deploy` time. Use `Path(__file__)` to anchor it reliably.

**Image defined in a shared `src/` monorepo (recommended for Pattern A):**

```python
# src/my_app/tasks/etl_tasks.py
PROJECT_ROOT = Path(__file__).parent.parent.parent.parent  # -> project_root/

etl_env = flyte.TaskEnvironment(
    name="etl",
    image=flyte.Image.from_debian_base()
        .with_uv_project(
            pyproject_file=PROJECT_ROOT / "pyproject.toml",
            extra_args="--only-group etl",
        )
        .with_code_bundle(),
)
```

Both the ETL and ML environments point to the same `pyproject.toml` but install different dependency groups. The `extra_args` string is included in the image hash, so they produce separate, independent images.

**Image defined in a child package (Pattern B):**

```python
# my_app/src/my_app/env.py
MY_APP_ROOT = Path(__file__).parent.parent.parent        # -> my_app/
MY_LIB_PKG = MY_APP_ROOT.parent / "my_lib" / "src" / "my_lib"

env = flyte.TaskEnvironment(
    name="my_app",
    image=flyte.Image.from_debian_base()
        .with_uv_project(pyproject_file=MY_APP_ROOT / "pyproject.toml")
        .with_source_folder(MY_LIB_PKG)
        .with_code_bundle(),
)
```

Each app defines its own image pointing to its own `pyproject.toml` and `uv.lock`. Sibling libraries are baked into the image via `with_source_folder()`. Fully independent images with no shared lock file.

## Section 7: Monorepo Patterns

Two patterns cover most cases. **Shared Lockfile** (Pattern A): all packages live under one `src/` directory with a single `pyproject.toml` and `uv.lock`; tasks install different subsets via dependency groups. **Independent Packages** (Pattern B): each package has its own `pyproject.toml` and `uv.lock`; fully independent image builds.

### Pattern A: Shared Lockfile, Flat `src/` Layout (recommended)

Use when: multiple packages are developed together, should share a dependency graph, and tasks need different subsets of a common dep pool.

```
workspace_root/
+-- pyproject.toml         <- defines dependency groups
+-- uv.lock                <- one lockfile for everything
+-- src/
    +-- my_app/
    |   +-- __init__.py
    |   +-- main.py
    |   +-- tasks/
    |       +-- envs.py
    |       +-- etl_tasks.py
    |       +-- ml_tasks.py
    +-- lib_a/
    |   +-- __init__.py
    +-- lib_b/
        +-- __init__.py
```

Put all packages directly under `src/`. `lib_a` and `lib_b` do not need their own `pyproject.toml` — they are importable because `src/` is `root_dir` and on `sys.path`.

`workspace_root/pyproject.toml`:

```toml
[project]
name = "my-app"
version = "0.1.0"
requires-python = ">=3.11"
dependencies = ["flyte>=2.0"]

[build-system]
requires = ["uv_build>=0.9,<0.10"]
build-backend = "uv_build"

[tool.uv]
package = true

[dependency-groups]
etl = ["pandas", "pyarrow"]
ml  = ["torch"]
dev = ["pytest", "ruff"]
```

Only list external PyPI packages in your dependency groups — not `lib_a` or `lib_b`. Those travel via the code bundle.

Per-task images using dependency groups:

```python
# src/my_app/tasks/envs.py
PROJECT_ROOT = Path(__file__).parent.parent.parent.parent  # -> workspace_root/

etl_env = flyte.TaskEnvironment(
    name="etl",
    image=flyte.Image.from_debian_base()
        .with_uv_project(
            pyproject_file=PROJECT_ROOT / "pyproject.toml",
            extra_args="--only-group etl",
        )
        .with_code_bundle(),
)
```

Entry point `root_dir`:

```python
# src/my_app/main.py
SRC_DIR = Path(__file__).parent.parent  # -> src/
flyte.init_from_config(root_dir=SRC_DIR)
```

Set `root_dir = src/` so the code bundle covers all packages under `src/`. Import `lib_a` and `lib_b` at the top level of your task modules so they are captured by the default `loaded_modules` bundling. For `copy_style="none"` production deploys, use `flyte.deploy()` from your entry point script.

### Pattern B: Independent Packages (independent lockfiles)

Use when: packages have different release cadences, are deployed independently, or should be fully self-contained per-package.

```
repo_root/
+-- pyproject.toml         <- dev-only convenience (optional)
+-- uv.lock                <- dev-only lockfile
+-- my_app/
|   +-- pyproject.toml     <- lists external PyPI deps + my-lib as editable path dep
|   +-- uv.lock            <- deployment lockfile (hermetic)
|   +-- src/my_app/
|       +-- __init__.py
|       +-- env.py
|       +-- main.py
|       +-- tasks.py
+-- my_lib/
    +-- pyproject.toml
    +-- src/my_lib/
        +-- __init__.py
        +-- utils.py
```

`my_app/pyproject.toml`:

```toml
[project]
name = "my-app"
version = "0.1.0"
requires-python = ">=3.11"
dependencies = ["flyte>=2.0", "my-lib"]

[tool.uv.sources]
my-lib = { path = "../my_lib", editable = true }

[build-system]
requires = ["uv_build>=0.9,<0.10"]
build-backend = "uv_build"
```

`my-lib` is declared as an editable path dep so the image builder can locate its source during `uv sync`. Only PyPI-installable packages go in `dependencies` otherwise.

Image definition:

```python
# my_app/src/my_app/env.py
MY_APP_ROOT = Path(__file__).parent.parent.parent    # -> my_app/
MY_LIB_PKG = MY_APP_ROOT.parent / "my_lib" / "src" / "my_lib"

env = flyte.TaskEnvironment(
    name="my_app",
    image=flyte.Image.from_debian_base()
        .with_uv_project(pyproject_file=MY_APP_ROOT / "pyproject.toml")
        .with_source_folder(MY_LIB_PKG)   # bakes my_lib into /root/my_lib/ in the image
        .with_code_bundle(),
)
```

`with_source_folder(MY_LIB_PKG)` copies the `my_lib` package directory into the image at `/root/my_lib/`. This is necessary because the editable install's `.pth` file points to a path that only exists during the image build stage, not in the final image. With `/root` on `PYTHONPATH`, `import my_lib` finds `/root/my_lib/__init__.py` at runtime.

The `my_lib` layer is part of the image hash, so the image rebuilds when `my_lib` changes — correct behavior for a dependency. Your `my_app` code changes don't affect this layer; they travel via the code bundle.

Entry point:

```python
# my_app/src/my_app/main.py
MY_APP_ROOT = Path(__file__).parent.parent.parent  # -> my_app/
SRC_DIR = MY_APP_ROOT / "src"                      # -> my_app/src/

flyte.init_from_config(root_dir=SRC_DIR)
# root_dir covers only my_app; my_lib is baked into the image
```

The `my_app/uv.lock` is the deployment lockfile. The SDK auto-detects it from the same directory as `pyproject_file`. The root `repo_root/uv.lock` is for local development only.

## Section 8: The Full Build Path (Production)

For production deployments where you need immutable, self-contained images:

```python
flyte.deploy(my_env, copy_style="none", version="1.2.3")
```

`with_code_bundle()` on the image resolves to a `COPY` instruction. The source is baked in. No code bundle at runtime. The image is fully self-contained.

Version management: use a deterministic string, a git commit SHA, a git tag, a CI build number. Avoid auto-generated strings for production images so you can trace which code is in which image.

**Do not use `install_project` mode for this.** `install_project` copies the entire project directory into the build context and hashes all of it. Every code change triggers a full image rebuild. `with_code_bundle()` + `copy_style="none"` is more surgical: only the files you select (via `loaded_modules` or `all`) are in the image, and the image hash reflects only those files.
