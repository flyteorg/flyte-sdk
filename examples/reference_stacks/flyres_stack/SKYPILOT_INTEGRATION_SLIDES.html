The Model Factory

Foundation model teams don't run one training job — they run an assembly line of interdependent stages that must iterate faster than humans can manually coordinate.

What a factory must handle

  • Train → checkpoint → evaluate → refine data → retrain — with dependencies, not handoffs
  • Combinatorial experiments: architecture ablations, data mixes, hyperparameter sweeps
  • Reproducibility: link every checkpoint back to dataset version, config, and infra used
  • Heterogeneous scheduling: CPU data jobs, GPU training, lightweight inference for evals
  • Fast feedback: schedule a new sweep in minutes, not days of manual pipeline wrangling
"Instead of manually triggering pipeline steps, describe an experimental configuration and the factory handles everything else — even triggering downstream steps automatically."

— Poolside, Introducing the Model Factory

How factories evolve

Six iterations of complexity

  • Training → automated evals → RL / code execution
  • Architecture ablations → data refinement → data mixing
  • Each stage adds dependencies the orchestrator must understand
The bottleneck shifts: it's no longer "can we train?" but "can we coordinate training, eval, data, and infra at scale without building a god-orchestrator?"

Layered orchestration stack

Separate concerns — macro workflow, cloud provisioning, container hosting, distributed compute — so teams scale from local experimentation to tens of thousands of remote GPUs.

Flyte 2
Macro

Workflow state, lineage, triggers. Dictates the what and when — checkpoint-triggered evals, semantic cache, typed artifacts. Built for compute pipelines with GPUs and terabyte-scale outputs — not just table freshness.

SkyPilot
Broker

Multi-cloud provisioning. Takes resource requests ("64 H100s"), finds cost-effective zones across AWS/GCP/Azure/K8s, provisions instances, queues jobs, handles spot recovery and failover.

Kubernetes
Base

Container orchestration. Node lifecycles, networking, storage — the foundation SkyPilot provisions against or runs on top of in BYOC clusters.

Ray / PyTorch
Micro

Distributed compute. Shards model state and data across GPUs on provisioned nodes. Intra-cluster fault tolerance without the macro-orchestrator micromanaging every worker.

Why Flyte at the macro layer? Dagster excels at asset catalogs and warehouse partitions; Flyte excels when the factory is a pipeline of compute steps — ETL, training, eval, and serve in one durable run graph. See slides 4–5.

Why Flyte for the Model Factory

Teams evaluating a single orchestrator for both data engineering and ML lifecycles — Dagster is strong on catalogs; Flyte is AI/ML-native by design.

Flyte 2 — compute & ML pipelines

  • First-class GPU Resources and task plugins (Ray, PyTorch Elastic, vLLM, SkyPilot)
  • Distributed training stays inside the orchestrator — not an external Ray job to babysit
  • flyte.File / Dir for TB checkpoints — references, not repeated rsync
  • Task-level semantic cache from input + code hash — skip redundant prep and training
  • Container Image defined next to tasks — built for CUDA rebuild cycles
  • One run ID links dataset → weights → eval metrics → deploy — full replay fidelity
  • SkyPilot as a plugin task, not an opaque sky launch subprocess in an op

Dagster — asset catalog & analytics DE

  • Asset-centric: tables, partitions, freshness, asset checks
  • Mature partition UI and backfill for warehouse pipelines
  • Distributed training via external Ray/K8s launched from assets
  • I/O managers + path strings for large artifacts — lineage is manual at TB scale
  • Execution environment configured on executor, not co-located with ML code
  • SkyPilot typically wrapped in an op — same subprocess risk
  • Two operational surfaces: Dagster run + external training job

One orchestrator for ML/AI-native and macro data pipelines

Flyte isn't ML-only. The same task graph that runs GPU training also owns curation, batch ETL, and warehouse loads — one trigger model, one artifact store, zero cross-system handoffs.

Data orchestration patterns Flyte owns natively

  • Multimodal curation: parallel modality preprocessing via asyncio.gather / fanout tasks
  • Batch ETL: S3 extract → transform → warehouse load with partitioned inputs
  • Cron schedules: Trigger.cron on the load task — same trigger model as GPU training jobs
  • Idempotent reruns: cache="auto" skips extract/transform when inputs unchanged
  • Typed artifact passing: flyte.File / Dir — platform handles blob movement between steps
The compounding advantage: data prep and model training share one lineage graph, one cache layer, and one failure-recovery surface. Debugging a bad model traces back through the ETL that produced its dataset — in a single UI.

Model Factory lifecycle

Three phases every frontier lab runs on repeat. Each phase maps to a distinct layer in the stack.

1

Training

Flyte triggers on a materialized dataset → SkyPilot provisions the GPU fleet → Ray/PyTorch distributes training → metrics stream to W&B or internal dashboards. Spot preemption handled at the SkyPilot or Ray layer.

2

Benchmarking & evaluation

Flyte detects a new checkpoint asset → automatically triggers eval workflows → SkyPilot spins inference instances → Ray Serve or actors parallelize thousands of prompts → results materialize as scored assets (MMLU, HumanEval, custom suites).

3

Lineage & experiment tracking

W&B holds granular metrics (loss curves, hyperparams). Flyte holds the lineage graph — if eval degrades, trace back to exact dataset version, SkyPilot config, and training run that produced the checkpoint.

Why this matters for the plugin

  • Flyte owns cross-stage dependencies and the audit graph
  • SkyPilot owns where each stage runs and how infra is recovered
  • Without the bridge, teams glue the two with subprocesses and lose lineage

The problem

Where the stack breaks today

  • Macro-orchestrator and cloud broker (SkyPilot) are disconnected — manual glue code
  • Teams split DE (Dagster) and ML (Flyte/Ray) — two runbooks, broken lineage at the handoff
  • Checkpoint → eval dependencies exist in notebooks, not in a durable lineage graph
  • GPU fleets span multiple clouds and clusters with no unified scheduling story
  • Experiments can't be reproduced: infra config lost when jobs are launched ad hoc
  • Iteration speed capped by manual sky launch + polling, not programmatic sweeps
One orchestrator should cover ETL and the Model Factory — Flyte for macro workflow, SkyPilot for multi-cloud provisioning. No god-orchestrator, no hybrid ops tax.

Each tool owns what it's best at

Flyte 2 — the what

  • DAGs, typed I/O, lineage
  • Semantic cache & replay
  • Cron & event triggers
  • Audit trail & retries

SkyPilot — the where

  • 20+ clouds + K8s / Slurm / VMs
  • Spot recovery & autostop
  • Managed jobs & sky serve
  • Multi-region failover bidding

Developer experience: one Python file

Spot-aware, multi-region GPU job — cron-triggered, retried, full Flyte lineage. No sky launch, no YAML, no polling. 

import flyte
from flyte import Image
from flyte.io import File
from flyteplugins.skypilot import SkyPilotJob

env = flyte.TaskEnvironment(
    name="batch_inference",
    image=Image.from_debian_base(name="scorer").with_pip_packages(
        "torch", "flyteplugins-skypilot",
    ),
    plugin_config=SkyPilotJob(
        accelerators="A100:1",
        use_spot=True,
        regions=["us-west-2", "eu-west-1", "ap-northeast-1"],
        job_recovery="EAGER_NEXT_REGION",
    ),
)

@env.task(
    retries=flyte.RetryStrategy(count=2),
    triggers=flyte.Trigger.cron("0 */6 * * *"),
)
async def batch_score(shard: File) -> File:
    from scorer import run
    return run(shard)

The connector: create → poll → complete

class SkyPilotConnector(AsyncConnector):
    task_type_name = "skypilot"

    async def create(self, task_template, inputs, ...):
        sky_task = _build_sky_task(spec, task_template, inputs)
        sky.api.login(endpoint=api_server, token=token)

        if task_template.type == "skypilot_job":
            request_id = sky.jobs.launch(sky_task, ...)
        else:
            request_id = sky.launch(task=sky_task, detach_run=True)

        job_id, handle = sky.stream_and_get(request_id)
        return SkyPilotJobMetadata(job_id=job_id, ...)

    async def get(self, resource_meta, ...):
        status = sky.job_status(...) # or sky.jobs.queue
        return Resource(phase=..., log_links=[dashboard_link])

    async def delete(self, resource_meta, ...):
        sky.jobs.cancel(...) # or sky.cancel

Execution lifecycle

  1. Flyte materializes inputs to object store
  2. Connector calls sky.launch or sky.jobs.launch
  3. SkyPilot provisions GPU pod/VM on target cloud
  4. Connector polls status → Flyte UI shows phase + logs
  5. Outputs written to Flyte object store; lineage stored
  6. SkyPilot autostop tears down cluster asynchronously
Identical lifecycle to Databricks connector — only the upstream verbs differ.

@prep_env.task(cache="auto")
async def prepare(raw: Dir) -> Dir: ...

@train_env.task(plugin_config=SkyPilot(accelerators="H100:8", use_spot=True))
async def finetune(dataset: Dir, epochs: int) -> Dir: ...

@eval_env.task(plugin_config=SkyPilotJob(job_recovery="EAGER_NEXT_REGION"))
async def evaluate(weights: Dir, eval_set: Dir) -> File: ...

@prep_env.task(triggers=flyte.Trigger.cron("0 2 * * *"))
async def main(raw: Dir, eval_set: Dir) -> File:
    dataset = await prepare(raw)
    weights = await finetune(dataset, epochs=3)
    return await evaluate(weights, eval_set)

flyte.deploy(main, serve_env)  # SkyServeAppEnvironment for inference

Completing the factory stack

One orchestrator, not two

"Avoid the common hybrid: Dagster for warehouse ETL, Flyte for ML — connected only by object-storage URIs and two operational runbooks. Flyte covers DE and the Model Factory in one graph; SkyPilot extends it to multi-cloud GPUs."

Starting from Flyte

"Flyte already orchestrates the factory workflow. The SkyPilot plugin lets training, eval, and serve stages reach spot capacity anywhere — with every run traced back through the lineage graph."

Deployment options

  • Plugin: OSS (Apache 2.0) in flyte-sdk monorepo
  • Self-hosted: Run Flyte and SkyPilot API server in your own infra
  • Managed control plane: Hosted Flyte + bring-your-own SkyPilot API server
  • Support scope: Connector, IDL, SDK + best-effort upstream SkyPilot triage

What each side owns

  • Flyte: orchestration, data, lineage, triggers
  • SkyPilot: provisioning, spot recovery, multi-cloud bidding
  • Cloud credentials stay with SkyPilot — Flyte never sees them

SkyPilot bootstraps an entire Union data plane on demand. Autostop tears it down. Zero idle cost.

How it works

1

SkyPilot provisions VMs

Finds cheapest spot capacity across clouds. Temporarily forwards cloud creds to remote nodes — same mechanism already used for data-source access.

2

Bootstrap k3s + Union data plane

sky local up --ips ... deploys k3s on the provisioned VMs. Helm installs FlytePropeller + Union operator. GPU Operator set up automatically if GPUs present.

3

Data plane registers with control plane

Union operator phones home, adds cluster credentials to FlyteAdmin. Control plane routes work to the new data plane — same object store bucket throughout.

4

SkyPilot autostops → deregister

On idle, SkyPilot tears down VMs. Union operator drains and deregisters the cluster. Control plane routes subsequent work elsewhere.

Path to the ephemeral data plane vision

Most building blocks already exist. Two gaps need closing before the vision is production-ready.

What already exists

✓ SkyPilot HA API server
PostgreSQL-backed, stateless pod, rolling upgrades — Haytham's original concern resolved
✓ k3s bootstrap from VMs
sky local up --ips <hosts> deploys k3s, GPU Operator, kubeconfig in one command
✓ Flyte multi-data-plane routing
FlyteAdmin supports clusterConfigs + labelClusterMap for multi-cluster dispatch
✓ Union BYOC operator model
Union operator already manages data plane lifecycle from within the cluster
✓ Credential forwarding
SkyPilot temporarily forwards cloud creds — gives ephemeral cluster access to shared object store

What needs to be built

Gap 1 — Dynamic cluster registration

FlyteAdmin currently requires static clusterConfigs in Helm — a restart to add/remove a cluster. Needs a runtime registration API (or Union operator webhook) so an ephemeral data plane can register itself on boot and deregister on drain without a control-plane reconfig.

Gap 2 — Bootstrap latency

k3s + FlytePropeller + Union operator install time must be short enough (~2–5 min) to be worth it for GPU workloads. Needs a pre-baked VM image (AMI / GCP image) with all dependencies installed, so bootstrap is just a cluster join + Helm upgrade — not a full install.

Zero idle cost: SkyPilot creates clusters on demand and destroys them when work is done. No standing GPU infrastructure to maintain or pay for between factory runs.
