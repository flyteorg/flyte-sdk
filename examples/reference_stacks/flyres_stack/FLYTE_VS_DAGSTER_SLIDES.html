Different mental models, same problem space

Both tools orchestrate data and compute. They differ fundamentally in what they treat as the primary unit.

Takeaway: Dagster excels when the team thinks in tables and assets (catalog, freshness, partitions). Flyte excels when the team thinks in pipelines of compute steps with large artifacts, GPUs, and reproducible reruns.

ETL and GPU training in one run graph

The FLYRES model factory starts with data. Both tools can run ETL. The difference: in Flyte the same graph that curates data continues into GPU fine-tuning — no scheduler boundary, no subprocess.

Flyte 2
# ETL tasks and GPU task — one graph
@env.task(cache="auto")
async def extract(partition_date: str) -> Dir:
    return Dir(path=f"s3://lake/dt={partition_date}")

@env.task
async def transform(raw: Dir) -> File:
    return File(path="s3://lake/cleaned.parquet")

# Continues into GPU fine-tuning — same graph,
# same lineage, no second scheduler
@train_env.task(
    plugin_config=SkyPilot(...),
    resources=Resources(gpu="A10G:4"),
)
async def finetune(dataset: Dir) -> Dir: ...
One orchestrator End-to-end lineage Typed artifact passing
Dagster
# ETL assets — catalog-first, great UI
@dg.asset(partitions_def=daily)
def extract(context) -> str:
    return f"s3://lake/dt={context.partition_key}"

@dg.asset
def transform(context, extract: str) -> str:
    return "s3://lake/cleaned.parquet"

# Training: a separate op / Ray job
# launched as a subprocess or external call —
# orchestration crosses a boundary here
@dg.asset
def finetune(context, transform: str) -> dict:
    import subprocess
    subprocess.run(["sky", "launch", "train.yaml"])
    return {"checkpoint": "s3://models/ckpt"}
Asset catalog Partition UI
The compounding advantage: Flyte's data prep and model training share one lineage graph, one cache layer, and one failure-recovery surface. A bad model traces back through the ETL that produced its dataset — in one UI.

Distributed GPU training: first-class vs. external

For multi-node PyTorch (FSDP / DDP), Flyte owns the process group lifecycle. Dagster delegates to Ray or a K8s job — two operational surfaces.

Flyte 2 — plugin owns the cluster
from flyteplugins.pytorch.task import Elastic

torch_cfg = Elastic(
    nproc_per_node=2, nnodes=2, max_restarts=3,
)
train_env = flyte.TaskEnvironment(
    name="fsdp-train",
    plugin_config=torch_cfg,
    image=Image.from_debian_base(
        python_version=(3, 12),
    ).with_pip_packages("torch==2.4.0"),
    resources=Resources(
        gpu="A10G:1", cpu=(4, 8),
        memory=("8Gi", "32Gi"),
    ),
)

@train_env.task
async def train(hyperparams: dict) -> dict:
    final_loss = train_loop(...)
    return {"checkpoint": "s3://...",
            "final_loss": final_loss}
  • Elastic restarts — orchestrator-managed
  • GPU resources declared alongside task
  • W&B run ID tied to Flyte run ID
Dagster — delegates to external runtime
# Training runs inside the asset body.
# Dagster does not own the process group.
@dg.asset(deps=["curate_dataset"])
def training_run(
    context: dg.AssetExecutionContext,
    curate_dataset: dict,
) -> dict:
    import wandb

    hyperparams = {"epochs": 3, "lr": 1e-3}
    wandb.init(project="dagster-experiments")
    final_loss = train_loop(...)
    wandb.log({"train_loss": final_loss})
    return {
        "checkpoint": "s3://models/ckpt",
        "final_loss": final_loss,
    }
  • No built-in PyTorch Elastic plugin
  • GPU spec lives in executor config, not code
  • Distributed setup: Ray job or K8s manifest
FLYRES takeaway: For scarce GPUs and multi-node PyTorch, Flyte's plugin model keeps orchestration and training in one system. Dagster remains a strong coordinator if Ray already exists, but you maintain two operational surfaces.

Data gravity: typed artifacts vs. path strings

Model weights and training datasets are measured in terabytes. How each orchestrator handles blob movement between steps has major operational impact.

Flyte 2 — platform stages blobs
from flyte.extras import Volume, volume_image

MODEL = "bert-base-uncased"

env = flyte.TaskEnvironment(
    name="hf-vol",
    image=volume_image(
        flyte.Image.from_debian_base().
        with_pip_packages("transformers"),
    ),
)

@env.task
async def cache_base_model() -> Volume:
    vol = Volume.empty(name="hf-weights")
    await vol.mount()
    model = AutoModel.from_pretrained(MODEL)
    model.save_pretrained("/workspace")
    return await vol.commit()
# Volume carries lineage — who produced it,
# which task version, which run ID.
Typed Volume Lineage in run graph Cache = input + code hash
Dagster — shared path assets
MODEL = "bert-base-uncased"
SHARED = "/shared/models/bert-base"

@dg.asset
def cache_base_model(
    context: dg.AssetExecutionContext,
) -> str:
    model = AutoModel.from_pretrained(MODEL)
    model.save_pretrained(SHARED)
    return SHARED

@dg.asset
def load_model(
    context,
    cache_base_model: str,
) -> dict:
    model = AutoModel.from_pretrained(
        cache_base_model,
    )
    return {"parameters": count_params(model)}
Asset catalog Shared FS required
Key difference: Flyte's Volume ties metadata index + blob chunks into the run graph (cache, fork, replay). Dagster achieves similar outcomes with shared volumes + path assets, but lineage across TB-scale weight trees is more manual.

Image as code vs. image as deployment config

CUDA base images take 20–40 minutes to build. Experimental layers (new transformers, peft version) should rebuild in seconds. How images are managed has a direct impact on ML iteration speed.

Flyte 2 — image declared next to task
from flyte import Image

# Slow CUDA base — rebuilt infrequently
base = (
    Image.from_debian_base(
        name="ml-base", python_version=(3, 12),
    )
    .with_pip_packages("torch==2.4.0")
    .with_apt_packages("cuda-compiler-12-4")
)

# Fast experimental layer — changes often
experimental = base.with_pip_packages(
    "transformers==4.42.0", "peft==0.11.0",
)

base_env = flyte.TaskEnvironment(
    name="base", image=base,
)
exp_env = flyte.TaskEnvironment(
    name="experimental", image=experimental,
)

@base_env.task
async def pretrain() -> str: ...

@exp_env.task
async def fine_tune(checkpoint: str) -> str: ...
Image is versioned with task Layered builds in SDK
Dagster — environment is deployment config
# dagster.yaml (conceptual)
# run_launcher:
#   module: dagster_k8s
#   class: K8sRunLauncher
#   config:
#     job_image: myregistry/ml-base:cuda12.4
#
# Bump experimental layer in CI:
# job_image: myregistry/ml-exp:transformers-4.42
  • Images are not a core Dagster concept
  • Execution environment lives in YAML config, not in code
  • Changing CUDA version requires infra change, not code change
  • Layer-aware build caching: bring your own Dockerfile / CI pipeline
Flyte's differentiator: image as code means CI can test "add peft to this environment" in the same PR that changes the fine-tuning task — no separate YAML PR, no container registry detour.

ML evaluation and metric gates

After training, run a benchmark suite, gate promotion on metrics, and register the artifact with full lineage back to training run and training dataset.

Flyte 2 — eval is a task in the same graph
base_env = flyte.TaskEnvironment(name="eval")

@base_env.task
async def evaluate_model(
    training_result: dict,
) -> dict:
    metrics = training_result["final_metrics"]
    return {
        **training_result,
        "accuracy": metrics.get("accuracy", 0.82),
        "f1_score": 0.80,
        "evaluation_status": "success",
    }

@base_env.task
async def register_model(
    evaluation: dict,
) -> dict:
    acc = evaluation["accuracy"]
    return {
        **evaluation,
        "registry": "text-classifier-v1",
        "deployment_ready": acc > 0.7,
    }
  • Eval run ID traces to training run ID
  • Gate logic in Python — flexible
  • Replay from eval without re-training
Dagster — asset_check for metric gates
@dg.asset(deps=[training_run])
def evaluation_metrics(
    context: dg.AssetExecutionContext,
    training_run: dict,
) -> dict:
    return {"accuracy": 0.82, "f1_score": 0.80}

# Asset checks: excellent for data quality gates
@dg.asset_check(asset=evaluation_metrics)
def accuracy_threshold(
    evaluation_metrics: dict,
) -> dg.AssetCheckResult:
    acc = evaluation_metrics["accuracy"]
    return dg.AssetCheckResult(
        passed=acc > 0.7,
        metadata={"accuracy": acc},
    )

@dg.asset(deps=[evaluation_metrics])
def registered_model(context) -> str:
    return "registry://text-classifier-v1"
  • Asset checks: clean UI for quality gates
  • Asset health status in Dagster+
Dagster's strength: @dg.asset_check is excellent for data-quality and metric gates in a catalog mental model. Flyte's strength: end-to-end run replay — same run ID ties training output URI, eval version, and deploy task.

Event-driven: file sensor (poll S3 → run task)

When new files land in object storage, kick off an ingestion task automatically.

Flyte 2 — polling app + task
task_env = flyte.TaskEnvironment(name="ingest")

@task_env.task
async def ingest_new_files(
    prefix: str, batch_id: str,
) -> str:
    return f"ingested:{prefix}:{batch_id}"

app = FastAPI()

@app.on_event("startup")
async def poll_s3():
    seen: set[str] = set()
    while True:
        for key in list_new_keys("s3://bucket/"):
            if key not in seen:
                seen.add(key)
                await flyte.run(
                    ingest_new_files,
                    prefix=key, batch_id=key,
                )
        await asyncio.sleep(60)

sensor_app = FastAPIAppEnvironment(
    name="s3-sensor", app=app,
)
Dagster — native @dg.sensor
@dg.asset
def ingest_new_files(context) -> str:
    cfg = context.op_config
    return f"ingested:{cfg['prefix']}"

ingest_job = dg.define_asset_job(
    "ingest_job", selection="ingest_new_files",
)

@dg.sensor(
    job=ingest_job,
    minimum_interval_seconds=60,
)
def s3_file_sensor(context):
    for key in list_new_keys("s3://bucket/"):
        yield dg.RunRequest(
            run_key=key,
            run_config={"ops": {"ingest_new_files": {
                "config": {"prefix": key},
            }}},
        )

defs = dg.Definitions(
    assets=[ingest_new_files],
    jobs=[ingest_job],
    sensors=[s3_file_sensor],
)
Dagster advantage here: @dg.sensor is a first-class primitive with deduplication via run_key, cursor tracking, and UI visibility. Flyte's polling app works but is more boilerplate to build and operate.

Lineage, caching, and the model factory vision

The Model Factory pattern — orchestrated prep → train → eval → serve — maps differently to each tool. Lineage and caching fidelity determine how fast teams can iterate.

The Flyte edge for model factories: every run carries an immutable lineage receipt — "this checkpoint was produced from this ETL run of this dataset version using this task code version." Reproducing or forking any point in the factory is a one-command operation.

When to choose which — for FLYRES-like teams

Both are excellent tools. The decision hinges on whether GPU compute orchestration or asset cataloging is the dominant pain point.

✦ Prefer Flyte 2

  • One orchestrator must cover DE ETL and GPU training without a subprocess boundary
  • Lineage and rerun fidelity across checkpoints, eval metrics, and serving are compliance- or incident-driven requirements
  • Large artifacts and semantic caching dominate cost and iteration time
  • Already on or targeting Kubernetes (Union or self-hosted Flyte 2)
  • CUDA image iteration speed matters — want image-as-code next to tasks

✦ Prefer Dagster

  • Organization is already asset-centric for analytics/DE with Dagster+ catalog, checks, and partitions
  • Workloads are primarily SQL / dbt / Spark-style pipelines with occasional ML jobs launched externally
  • Partition backfills and data contracts matter more than multi-node PyTorch orchestration
  • Native sensor primitives and Declarative Automation fit the event model better

⇄ Hybrid (common in practice)

  • Dagster for warehouse-centric ETL; Flyte for ML factory stages — connected via object storage URIs
  • Matches SPEC's "avoid two orchestrators" tension: workable, but operational overhead remains
  • Evaluate total cost: two infra stacks, two on-call surfaces, two deployment pipelines
FLYRES recommendation: If the primary workload is an AI model factory (curate → train → eval → serve with GPU at every stage), Flyte's unified graph and first-class plugin model justify the learning curve over Dagster's asset-first paradigm.
